# Режим сервиса Turnable &nbsp;·&nbsp; [🇬🇧 EN](SETUP.md)
Режим сервиса предоставляет уровень управления для запуска нескольких инстансов сервера или клиента Turnable. Он предоставляет протокол управления через Unix сокеты или TCP, позволяя создавать, запускать, останавливать и управлять экземплярами динамически без перезагрузки.

## Гайды для разных платформ
- [Настройка на Windows](WINDOWS_RU.md)
- [Настройка на Linux](LINUX_RU.md)

## Ключевые особенности
- Запуск нескольких инстансов сервера или клиента из одного сервиса
- Динамическое управление инстансами через CLI клиент или программный API
- Опциональное шифрование и аутентификация с использованием ML-KEM-768
- Автоматическое сохранение и автоматический перезапуск инстансов
- Структурированное логирование с потоковой передачей в реальном времени
- Unix или TCP сокеты

## 1. Генерирование ключей шифрования (опционально)
Для безопасной настройки с аутентификацией генерируйте пары ключей ML-KEM-768:
```bash
./turnable config keygen
```

Вы получите вывод вроде:
```
priv_key=whH/S/GPFJ37zGv8n...
pub_key=BWEx0ygunbFJFCrIN...
```

Сервис будет использовать эти ключи для шифрования. Генерируйте отдельные пары ключей для сервера и для каждого клиента, которому нужен доступ.

## 2. Создание конфигурации сервиса
Создайте файл `service.json`:
```json
{
  "unix_socket": "/tmp/turnable.sock",
  "listen_addr": "127.0.0.1:9000",
  "pub_key": "BWEx0ygunbFJFCrIN...",
  "priv_key": "whH/S/GPFJ37zGv8n...",
  "allowed_keys": [
    "client_pub_key_1_base64",
    "client_pub_key_2_base64"
  ],
  "persist_dir": "/var/lib/turnable/instances"
}
```

### Поля конфигурации
| Поле           | Обязательно | Описание                                                                   |
|----------------|-------------|----------------------------------------------------------------------------|
| `unix_socket`  | Нет         | Путь к Unix сокету для локальных клиентов (например, `/tmp/turnable.sock`) |
| `listen_addr`  | Нет         | TCP адрес и порт (например, `127.0.0.1:9000` или `0.0.0.0:9000`)           |
| `pub_key`      | Нет         | Публичный ключ ML-KEM-768 сервера для аутентификации (base64)              |
| `priv_key`     | Нет         | Приватный ключ ML-KEM-768 сервера для расшифровки (base64)                 |
| `allowed_keys` | Нет         | Список разрешенных публичных ключей клиентов (base64)                      |
| `persist_dir`  | Нет         | Директория для автозагрузки и сохранения конфигураций экземпляров          |

> [!NOTE]
> Должно быть установлено хотя бы одно из `unix_socket` или `listen_addr`. Если пара ключей не предоставлена, сервис работает без аутентификации. Если аутентификация включена, требуется `allowed_keys`.

## 3. Запуск сервиса
```bash
./turnable service server -c service.json -p /var/lib/turnable/instances
```

Доступные флаги:
```
-c, --config string   путь к JSON файлу конфигурации сервиса (по умолчанию "service.json")
-p, --persist string  директория для сохранения конфигураций экземпляров для автозагрузки
-V, --verbose         включить подробное логирование отладки
```

## 4. Подключение с CLI клиентом
В другом терминале запустите CLI клиент:
```bash
./turnable service client --unix /tmp/turnable.sock
```

Или подключитесь через TCP с аутентификацией:
```bash
./turnable service client \
  --address 127.0.0.1:9000 \
  --pub-key client_pub_key_base64 \
  --priv-key client_priv_key_base64
```

Доступные флаги:
```
-u, --unix string      путь к Unix сокету для подключения
-a, --address string   TCP адрес и порт для подключения
-p, --pub-key string   публичный ключ ML-KEM-768 для аутентификации (base64)
-k, --priv-key string  приватный ключ ML-KEM-768 для аутентификации (base64)
```

## 5. Управление экземплярами в CLI
После подключения к сервису вы можете использовать команды. Напишите `help` для списка команд.

## 6. Пользовательские клиенты сервиса
Для интеграции с вашими собственными инструментами вы можете реализовать пользовательский клиент. См. [service.proto](/pkg/service/proto/service.proto) и [encryption.go](/pkg/service/encryption.go) для спецификации протокола и шифрования.
